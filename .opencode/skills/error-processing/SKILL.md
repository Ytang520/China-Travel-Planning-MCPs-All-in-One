# 错误处理

## 默认模式

使用半自动排障模式：

1. 先调查，不急于改动。
2. 如果原因明确、修复很小且风险低，可以直接做最小修改并验证。
3. 如果修复涉及密钥、凭证、依赖升级、破坏性命令、架构选择、大范围重构或不明确取舍，先停下来，给用户 2-3 个方案选择。

## 安全规则

- 不输出密钥、token、API key、cookie、完整环境变量或私人账号数据。
- 如果日志或配置片段里出现敏感值，必须打码后再展示。
- 除非用户明确批准，不运行破坏性命令。
- 编辑文件前优先做只读探索。
- 修复范围只围绕当前失败行为，避免顺手重构。

## 处理流程

当出现错误时：

1. 记录原始错误信息、触发命令、工作目录和相关近期改动。
2. 判断错误类型：
   - 配置错误
   - 依赖或环境错误
   - 构建或类型检查错误
   - 运行时错误
   - MCP 连接或鉴权错误
   - MCP 工具 schema 或返回格式错误
   - 网络或远端服务错误
   - 未知错误
3. 先检查最相关的文件：配置、入口文件、堆栈指向文件、provider 定义和近期编辑。
4. 提出最多 3 个可能原因，并说明每个原因的证据。
5. 用最小且安全的验证动作确认最可能的原因。
6. 只有在原因明确且改动低风险时，才直接修复。
7. 用最窄范围的命令或诊断验证问题已解决。
8. 简洁汇报结果。

## MCP 错误参考

当遇到 MCP 连接、鉴权、工具 schema 或返回格式问题时：

1. 读取本 skill 目录下的 `mcp-error-references.json`。
2. 优先根据每个参考源的 `description` 和 URL `usage` 做语义匹配。
3. `triggers` 只作为可选别名，用于匹配 provider id、环境变量名、工具名和中文名称。
4. 如果没有 trigger 命中，根据错误上下文和短描述选择最可能的 1-2 个参考源。
5. 如果仍无法判断 provider，继续按通用 MCP 排障流程处理，并说明不确定性。
6. 必要时访问匹配到的公开参考网址，核对当前连接方式、鉴权要求、工具能力或返回格式说明。

## 汇报格式

使用下面的简洁结构：

```text
错误摘要:
- ...

已检查:
- ...

判断:
- ...

下一步:
- 已直接修复并验证 / 需要你选择方案
```
